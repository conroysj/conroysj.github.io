Deciding on a Database

05 May 2015

On the heels of a great talk by the CTO of NPM - Laurie Voss, my thesis project team at Makersquare needed a database. Thankfully, the talk and our studies have informed us well, and after some short deliberation, we decided to use PostgreSQL.

Relational databases like PostgreSQL and MySQL (aka databases which use structured query language) use binary code to store information. This allows for speedy lookup times, in addition to efficient queries that can compare different, variable datasets. There are some limitations: namely, the structure of your dataset is most easily architechted and pliable at its initiation/at the outset. Additionally, entries are space-limited by a pre-defined integer/character specification and count.

An SQL database made most sense for our project because we wanted to be able to make comparative queries on the fly. Having a leaderboard that involved significant data queries/comparisons between users would have been difficult using a NoSQL object that consists of stored JSON objects, especially if we only wanted the top scorers, for instance. In an SQL set up, we can create quickly query the table that keeps track of the value of interest, rather than iterating through each user's JSON store in order to find the value of interest and compare going forward.

Nonrelational databases/NoSQL like Firebase allow for a quick storing of data and quick lookup, only when not seeking to compare datasets / identify relationships between them. They store data in no particular order, but the manner by which data is structured (ie the parameters, etc.) may be iterated on as desired. Additionally, unlike relational databases, they are not restricted by a pre-defined integer/character specification and count.

We considered other databases, all of which have their own unique set of advantages and disadvantages.

Sean Conroy
